# 貢獻指南

感謝你對本專案的關注。本合集致力於為中文使用者提供**經過驗證的** OpenClaw 真實用例——幫讀者找到「AI 智能體能改善生活的方式」，照著做就能跑通。

我們歡迎以下貢獻：

- **提交新用例**（國內生態原創 或 國際用例的國內適配）
- **改進現有用例**（補充實踐經驗、修正過時資訊）
- **改進翻譯或表述**（讓已有內容更準確、更自然）

---

## 收錄標準

用例必須同時滿足以下三條才會被收錄：

| 條件 | 說明 |
|------|------|
| **真實跑通** | 有端到端執行證據——設定步驟、指令、可複現的輸出，不收「想法帖」 |
| **多源驗證** | 至少有「官方文件/倉庫」+「社群實踐者內容」兩個獨立來源 |
| **可操作** | 有具體步驟和可複製的提示詞/設定，讀者照做能複現 |

不符合標準的內容不會被合併，但歡迎在 Issue 中討論。

---

## 用例類型

### 1. 國內生態原創用例

專為中國工具生態設計的用例（飛書、釘釘、企業微信、小紅書、AKShare 等）。

- 檔案放在 `usecases/` 目錄，檔案名稱使用 `cn-` 前綴，如 `cn-feishu-ai-assistant.md`
- 在 README 的 **中國特色用例** 表格中新增一行

### 2. 國際用例的國內適配

在已有國際用例基礎上，增加國內平台替代方案。

- **不修改原有內容**，在檔案底部新增 `## 中國用戶適配` 章節
- 檔案標題下方新增提示行：`> 含國內適配：XX / YY / ZZ`
- 在 README 的 **中國特色用例** 表格中新增一行，名稱標注「（適配）」
- 同時保留在下方對應分類表格中，名稱標注「（國內適配）」

### 3. 改進現有用例

補充實踐經驗、修正過時資訊、改進表述。直接修改對應檔案即可。

---

## 用例檔案格式

每個用例檔案按以下結構組織（根據實際情況可靈活調整）：

```markdown
# 用例名稱

簡短描述痛點和這個用例做什麼（2-3 句話）。

## 它能做什麼

列出核心功能點。

## 所需技能

需要安裝的 OpenClaw 技能/外掛，注明來源。

## 如何設定

分步驟說明設定方法，包含可複製的提示詞和指令。

## 實用建議

踩過的坑、最佳實踐、注意事項。

## 相關連結

參考文件、工具倉庫、社群文章等。
```

**要點：**

- 提示詞（prompt）建議保留英文原文，上方新增中文說明
- 程式碼區塊中的注解翻譯為中文，程式碼和指令保留英文
- 技術術語首次出現時括號注解，如 `webhook（網路鉤子）`
- 涉及社群媒體自動化的用例，必須包含平台風控提醒
- 不硬編碼 API 金鑰，憑證透過環境變數傳遞

---

## 提交流程

1. 從 `main` 分支建立新分支：`feat/用例名稱`
2. 在 `usecases/` 下建立或修改用例檔案
3. 更新 README：
   - 在對應表格中新增條目
   - 更新頂部用例計數徽章
4. 提交 PR 到 `main` 分支，標題使用中性描述

分支命名規範：`feat/xxx`（新功能）、`fix/xxx`（修正）、`improve/xxx`（改進）

---

## 不接受的內容

- 未經實際測試的用例（AI 生成但沒有跑通的不算）
- 加密貨幣相關用例
- 個人微信協議方案（封號風險極高）
- 硬編碼 API 金鑰或憑證

---

## 引用規範

- 優先引用官方文件和成熟開源專案
- 社群技能/外掛需注明來源，讓讀者自行判斷風險
- 不推薦未經驗證的社群外掛（僅有少量 star 的倉庫不算驗證）
- 商業 SaaS 產品不在正文推薦
